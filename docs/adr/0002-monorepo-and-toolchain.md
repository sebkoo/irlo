# 0002 — Monorepo and toolchain

- Status: accepted
- Date: 2026-07-10
- Deciders: Ben Koo

## Context and problem statement

Irlo spans a TypeScript backend (`server/`), a shared contracts package (`packages/contracts/`),
and a Swift iOS client (`apps/ios/`). An API change touches all three surfaces. We need a repo
layout where one reviewable PR can change a contract, the server, and the client together.
We also need two sub-decisions settled: how the Xcode project is defined and versioned, and how
tool versions are pinned identically on laptops and in CI. Today the repo holds only Stage-0
scaffolding plus canary tests; the layout must not impose orchestration cost we do not yet need.

## Decision drivers

- **D1 — Atomic cross-cutting change**: contract + server + client in a single PR and commit.
- **D2 — One source of truth for API shapes**: compile-time types and runtime validation from the same definition.
- **D3 — Overhead proportional to size**: two JS packages and one app today; no speculative machinery.
- **D4 — Reviewable, merge-safe diffs**: especially for Xcode project metadata, a notorious conflict magnet.
- **D5 — Reproducible toolchain**: one pin per tool, honored by `make bootstrap` locally and by CI.
- **D6 — Interview legibility**: a senior reviewer should grasp the layout and its wiring in minutes.

## Considered options

JS workspace and build:

1. **pnpm workspaces, no build orchestrator** (chosen)
2. pnpm workspaces + Turborepo from day one
3. Nx integrated monorepo
4. Single repo per service (polyrepo)

iOS project definition and dependencies:

5. **XcodeGen + Swift Package Manager, generated `.xcodeproj` gitignored** (chosen)
6. Committed `.xcodeproj` + SPM
7. CocoaPods
8. Tuist

## Decision outcome

**pnpm workspaces alone**, with Turborepo deliberately deferred; **XcodeGen + SPM** with the
generated project gitignored; **mise** pins node/pnpm and the **Brewfile** pins macOS-only tools.

- `pnpm-workspace.yaml` links `@irlo/contracts` into `@irlo/server` via `workspace:*`.
- **Internal-package pattern**: contracts export TypeScript source directly
  (`"exports": { ".": "./src/index.ts" }`). Consumers typecheck it with their own tsconfig.
  There is no build step, no `dist/`, and therefore no build orchestration to maintain.
- **Turborepo trigger (recorded now)**: adopt it when the workspace exceeds **3 packages** or a
  cold build/test of the JS workspaces exceeds **30 seconds**. Until then a task graph and remote
  cache would be configuration without payoff.
- The zod v4 schemas in `packages/contracts/` are the single source of truth for API shapes
  (per [ADR-0003](0003-backend-platform.md)). zod → OpenAPI generation is planned (Stage 1+), so
  REST documentation and client codegen derive from the same schemas rather than drifting copies.
- **XcodeGen**: `apps/ios/project.yml` is the human-reviewable source of truth; `Irlo.xcodeproj`
  is generated by `make bootstrap` and gitignored. Merge conflicts in generated project files are
  structurally impossible. Swift dependencies come via SPM (none yet; RxSwift and
  swift-snapshot-testing arrive per [ADR-0008](0008-ios-demo-client.md)).
- **One source of truth per tool**: `.mise.toml` pins node 24 and pnpm 10 (used locally and in
  CI); the Brewfile pins macOS-only tools (xcodegen, swiftlint, swiftformat, ffmpeg, gh).
  No version number appears in two files.

### Positive consequences

- A schema change, its server handler, and its client model land as one reviewable diff (D1).
- Contracts give end-to-end type safety with zero publish/version ceremony inside the repo (D2).
- Stage-0 wiring is a single YAML file and an `exports` map — nothing to explain away (D3, D6).
- `project.yml` diffs read like code review, not pbxproj archaeology (D4).
- `make bootstrap` reproduces the exact toolchain on any machine, including CI runners (D5).

### Negative consequences

- No task graph or remote cache: full re-runs as packages grow. Accepted; the Turborepo
  trigger above is the pre-committed exit.
- Source-exporting contracts means every consumer re-typechecks them, and publishing the package
  externally later would require adding a build step.
- Xcode-side project edits are lost on regeneration; every project change must go through
  `project.yml`. This is deliberate but surprises Xcode-native contributors.
- Contributors need mise and Homebrew before `make bootstrap` succeeds — a documented, real cost.
- pnpm's strict `node_modules` exposes phantom-dependency bugs in packages that forgot to declare
  imports; the fix (declare the dependency) is correct but occasionally annoying.

## Pros and cons of the options

Scores: `++` strong fit · `+` fit · `o` neutral · `–` poor · `––` disqualifying.

| JS workspace option | D1 atomic change | D2 one truth | D3 overhead today | D6 legibility |
|---|---|---|---|---|
| **pnpm workspaces (chosen)** | ++ | ++ (`workspace:*` source link) | ++ (one YAML file) | ++ |
| pnpm + Turborepo now | ++ | ++ | – (task graph + cache for 2 packages) | + |
| Nx | ++ | ++ | –– (plugins, generators, daemon) | o (framework-shaped repo) |
| Polyrepo | –– (multi-repo PRs) | –– (published versions, skew) | o | – (wiring lives in registries) |

| iOS option | D4 merge-safe review | D3 overhead today | D5 reproducibility |
|---|---|---|---|
| **XcodeGen + SPM, `.xcodeproj` gitignored (chosen)** | ++ (YAML diff; no generated-file conflicts) | + (one brew tool + generate step) | ++ (project derived, never drifts) |
| Committed `.xcodeproj` + SPM | –– (opaque pbxproj diffs, frequent conflicts) | ++ (zero extra tools) | + |
| CocoaPods | – (Pods churn, workspace mutation) | – (Ruby toolchain, `pod install`) | o; project entered maintenance mode (CocoaPods blog, 2024) — a shrinking bet |
| Tuist | ++ (Swift manifests) | – (heavier tool with its own caching/graph concepts) | + |

Tuist is the strongest rejected option: it wins for multi-target modular apps, which Irlo's
single-target demo client is not (D3). Revisit if `apps/ios` grows past roughly three frameworks.

## Future trends & implications

Over the next ~24 months, pnpm workspaces remain the default substrate for TypeScript monorepos,
and orchestrators keep converging on it rather than replacing it — our deferred-Turborepo posture
ages well. The zod ecosystem's convergence on Standard Schema makes the contracts package more
portable, and zod→OpenAPI generation increasingly commodity, strengthening the single-source bet.
On iOS, Apple keeps consolidating on SPM while CocoaPods winds down in maintenance mode, so the
SPM choice is the only direction with a future. XcodeGen is a thin, replaceable layer: if Apple
or Tuist ever makes `pbxproj` obsolete, migrating from `project.yml` is mechanical, not
architectural. The riskiest assumption is repo growth — if the planned React Native screen and AI
services multiply packages quickly, the 3-package/30-second trigger fires and Turborepo lands.
